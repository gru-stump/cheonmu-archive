# 천무 서사 기록 아카이브 설계

## 1. 목표

`천무_캐릭터_프로필.md`를 기반으로 천령과 무영의 관계 서사를 감상하고 설정을 관리하는 반응형 웹사이트를 만든다. 공개 사이트는 GitHub Pages에서 누구나 읽을 수 있고, 소유자는 로컬 편집기로 마크다운 콘텐츠와 갤러리 정보를 관리한다.

첫 버전의 성공 기준은 다음과 같다.

- 관계 변화 8단계를 기록일지 형식으로 자연스럽게 읽을 수 있다.
- 중요한 사건은 시네마틱 장면으로 전환해 감상할 수 있다.
- 인물 프로필, 설정 문서, 사건 자료, 커미션 이미지를 한곳에서 탐색할 수 있다.
- 확정 설정과 제안된 초안이 시각적·구조적으로 혼동되지 않는다.
- PC와 모바일에서 핵심 기능과 콘텐츠가 동등하게 제공된다.
- 소유자는 코드를 직접 편집하지 않고도 로컬 편집기에서 콘텐츠를 관리할 수 있다.

## 2. 범위

### 포함

- Vite, React, TypeScript 기반 정적 사이트
- React Router의 해시 기반 라우팅을 사용한 화면 이동과 GitHub Pages 직접 URL 대응
- 마크다운 중심 콘텐츠 저장
- 관계 변화 8단계 타임라인
- 핵심 사건 4개의 장면 재구성
- 천령·무영 상세 프로필과 페어 관계 문서
- 사건 자료와 설정 자료 아카이브
- 이미지 갤러리와 작품 크레딧
- 공개 빌드와 분리된 로컬 전용 편집기
- GitHub Pages 자동 배포 구성
- 콘텐츠 검증, 단위 테스트, 주요 사용자 흐름 테스트

### 제외

- 공개 사이트의 로그인과 편집
- 방문자 댓글 또는 투고
- 서버 데이터베이스
- 실시간 협업 편집
- 외부 검색 서버
- 첫 버전에서의 GitHub OAuth 연동

## 3. 기술 구조

### 공개 사이트

- Vite + React + TypeScript
- React Router의 `HashRouter`; GitHub Pages에 서버 재작성 규칙이 없어도 상세 화면 새로고침이 동작해야 함
- 빌드 시 마크다운을 읽어 정적 자산으로 묶는 콘텐츠 로더
- 필요한 범위에서만 애니메이션과 전체 화면 장면 전환 사용
- GitHub Actions로 빌드 후 GitHub Pages 배포

### 로컬 편집기

- `npm run editor`로 로컬 편집 서버와 편집 UI 실행
- 서버는 루프백 주소에만 바인딩해 외부 접속 차단
- 공개 프로덕션 번들에는 편집 경로와 파일 쓰기 코드를 포함하지 않음
- 콘텐츠 스키마 검증 후 마크다운과 메타데이터 파일 저장
- 삭제 전 확인과 복구 가능한 임시 보관 적용

공개 사이트와 편집기는 같은 콘텐츠 스키마를 사용하되, 읽기 계층과 쓰기 계층을 분리한다. 이후 인증 기반 웹 편집기를 도입할 때 화면 컴포넌트와 스키마를 재사용할 수 있다.

## 4. 정보 구조

상단 탐색은 `천무`, `기록철`, `아카이브` 세 항목으로 제한한다.

### 천무

- 페어 캐치프레이즈와 대표 이미지
- 두 인물의 짧은 소개
- 관계 핵심 문장
- 기록철 진입점

### 기록철

- 관계 변화 8단계의 세로 타임라인
- `전체`, `확정`, `초안` 필터
- 사건별 기록번호, 시점, 관계 단계, 상태, 요약, 대표 대사
- 관련 인물과 연결 기록
- 핵심 사건의 장면 재구성 진입점

### 아카이브

- 인물 프로필: 외형, 성격, 말투, 능력, 호칭, 상호 시선
- 사건 자료: 의료 기록, 현장 보고서, 미확정 단서
- 갤러리: 커미션, 페어틀, 전신, 동물화 및 향후 이미지
- 설정 문서: 페어 소개, 핵심 문장, 확정 설정, 추가 결정 항목

## 5. 시각 설계

기본 방향은 현장 보고서와 의료 차트를 닮은 기록일지다.

- 아이보리 문서, 검은 활자, 어두운 적색 도장을 기본 색으로 사용
- 천령은 탁한 녹색과 자주색, 무영은 흑색과 어두운 적색으로 구분
- 확정 콘텐츠에는 `확정 기록`, 제안 콘텐츠에는 `초안 기록` 또는 `열람 제한` 표식 사용
- PC에서는 좌측 분류철과 우측 문서를 함께 보여주는 조사 책상 구조
- 모바일에서는 상단 탭과 한 장씩 이어지는 기록 카드 구조
- 갤러리는 PC에서 비대칭 그리드, 모바일에서 1~2열 그리드 사용

핵심 장면을 열면 기록 문서가 어두운 시네마틱 화면으로 전환된다. 이 화면은 검정·회색·탁한 적색, 큰 타이포그래피, 짧은 서사와 대사에 집중한다. 닫으면 기존 기록 위치로 복귀한다.

## 6. 콘텐츠 모델

### 사건 기록

각 사건은 개별 마크다운 파일이며 다음 메타데이터를 가진다.

- 고유 ID와 기록번호
- 제목과 짧은 요약
- 관계 단계와 시점
- `confirmed` 또는 `draft` 상태
- 등장인물
- 태그와 관련 기록 ID
- 대표 대사
- 장면 재구성 제공 여부와 장면 순서

### 인물 프로필

천령과 무영을 별도 콘텐츠로 관리한다. 기본 정보, 외형, 성격, 말투, 능력, 대표 대사, 호칭, 상대를 보는 시선을 구조화한다. 원문 안의 무영 신장 `185cm`와 `189cm` 불일치는 사이트에 옮기기 전에 하나로 확정해야 하는 검증 오류로 취급한다.

### 갤러리

갤러리 항목은 다음 정보를 가진다.

- 작품 ID와 제목
- 이미지 경로와 대체 텍스트
- 작가명과 출처 링크
- 제작일
- 등장인물과 태그
- 공개 여부

현재 자산은 다음처럼 시작한다.

- `천무_전신.png`: 인물 파일과 갤러리
- `천무_페어틀.png`: 페어 개요와 갤러리
- `260719_grid (1).png`: 프로필·크레딧 참고 자료와 갤러리

### 설정 문서

페어 소개, 핵심 문장, 확정 설정, 추가 결정 항목을 별도 마크다운으로 관리한다. 원본 `천무_캐릭터_프로필.md`는 출처 자료로 보존한다.

## 7. 첫 버전 콘텐츠

- 관계 변화 8단계 전체
- 천령·무영 상세 프로필
- 페어 관계, 호칭, 상호 시선
- 확정 설정과 미확정 설정 목록
- 현재 이미지 3장으로 시작하는 갤러리
- 장면 재구성 4개: 첫 조우, 현장 동행, 균열, 귀환의 약속
- 첫 조우의 재난과 관계 전환 사건 등 새로 제안하는 내용은 모두 초안으로 표시
- 마지막 관계 상태는 감정이 명확하지만 관계명을 붙이지 않은 연인 직전

## 8. 상호작용

### 기록철

- 스크롤에 따라 사건 카드가 순서대로 나타남
- 상태 필터가 해시 URL의 검색 매개변수에 반영됨
- 사건을 열면 상세 문서로 이동
- 핵심 사건에서는 전체 화면 장면 재구성 제공
- 상세 화면에서 관련 인물과 연결 기록으로 이동 가능

### 조사 아카이브

- 자료 종류, 인물, 태그로 필터링
- 각 자료는 독립 URL을 가짐
- 존재하지 않는 기록은 아카이브 복귀 안내를 제공

### 갤러리

- 썸네일 그리드와 원본 비율 확대 보기
- 작품명, 작가명, 출처, 태그 표시
- 키보드 이전·다음 이동과 닫기 지원
- 비공개 항목은 프로덕션 빌드에서 제외

### 로컬 편집기

- 사건 생성·수정·삭제
- 확정/초안 상태, 인물, 태그, 관련 기록 지정
- 프로필과 설정 문서 수정
- 이미지 등록과 크레딧 입력
- 장면 대사와 순서 편집
- 저장 전 미리보기와 필수 항목 검증
- 변경된 파일 목록 표시

## 9. 데이터 흐름과 오류 처리

공개 빌드는 콘텐츠 로더가 마크다운과 이미지 메타데이터를 읽고 스키마를 검증한 뒤 화면용 모델로 변환한다. 필수 필드, 중복 ID, 끊어진 내부 링크, 존재하지 않는 이미지, 잘못된 상태값은 빌드 오류로 보고한다.

대체 가능한 정보가 누락된 경우 해당 카드에 안내를 표시하고 전체 사이트는 유지한다. 필수 식별자나 공개 이미지 경로처럼 안전하게 대체할 수 없는 오류는 배포 전에 빌드를 중단한다.

편집기는 저장 전에 동일한 스키마를 사용한다. 삭제한 콘텐츠와 이미지는 프로젝트 내부의 복구용 보관 위치로 이동하며, 사용자가 명시적으로 정리하기 전에는 영구 삭제하지 않는다.

## 10. 반응형과 접근성

- PC, 태블릿, 모바일에서 정보와 기능을 동일하게 제공
- 모바일에서 다단 패널은 탭과 단일 열 카드로 전환
- 키보드만으로 메뉴, 필터, 장면, 갤러리 조작 가능
- 명확한 포커스 표시와 충분한 색 대비
- 모든 콘텐츠 이미지에 의미 있는 대체 텍스트
- `prefers-reduced-motion` 사용자는 등장 및 전환 애니메이션 최소화
- 장면과 갤러리 모달은 포커스 고정, Escape 닫기, 닫은 뒤 원래 위치 복귀 지원

## 11. 테스트와 배포 검증

- 콘텐츠 스키마, 중복 ID, 내부 링크, 이미지 경로 검사
- 콘텐츠 로더와 주요 표시 컴포넌트 단위 테스트
- 기록 상태 필터, 사건 이동, 장면 열기·닫기 테스트
- 아카이브 필터와 갤러리 확대 보기 테스트
- 로컬 편집기의 생성·수정·검증·복구 가능한 삭제 테스트
- 대표 PC·태블릿·모바일 폭에서 시각 확인
- 키보드 탐색과 기본 접근성 검사
- GitHub Pages 기본 경로, 직접 URL 접근, 새로고침 검증
- 프로덕션 빌드에 편집기 코드와 비공개 갤러리 항목이 없는지 확인

## 12. 향후 확장

첫 버전 이후 별도 인증 계층을 도입하면 소유자 전용 웹 편집과 GitHub 저장소 커밋 기능을 추가할 수 있다. 이때 공개 읽기 화면과 콘텐츠 스키마는 유지하고, 로컬 쓰기 어댑터를 인증된 원격 저장 어댑터로 교체한다.

## 13. 확정 결정

- 결과물: 서사 감상과 설정 관리를 결합한 사이트

- 반응형: PC와 모바일 동등 지원
- 편집: 소유자 컴퓨터에서만 실행되는 로컬 편집기
- 결말: 연인 직전
- 콘텐츠 정책: 확정 내용과 제안 초안을 명확히 분리
